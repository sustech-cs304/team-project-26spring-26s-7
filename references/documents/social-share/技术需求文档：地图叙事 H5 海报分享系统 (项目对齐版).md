## 1. 项目概览

本项目为 TravelPin（鸿蒙旅行日记 App）的社交分享模块，目标是在 App 内将一段路线叙事打包为可限时访问的 H5 页面，并支持在系统分享面板、QQ、微信等场景中稳定分发。

本期交付范围：

- 路线图文信息采集、打包与上传
- 服务端接收数据、临时存储、组装 H5 展示页、生成限时签名 URL
- 前端对限时 URL 做可视化封装，支持 QQ / 微信分享与对应浏览器打开验证

本期**不包含**：

- PNG 海报导出
- 原生小程序卡片
- App 内重做一套独立地图回放引擎替代现有 TripReplayPage

与现有 replay 功能的关系：

> TripReplayPage (`feature/map-travel/pages/TripReplayPage.ets`) 已实现完整的路线回放动画系统，包括 5 阶段状态机（淡入 2s → 停留 3s → 淡出 1s → 相机移动 1.5s）、bearing/tilt 3D 视角、离散进度条、速度控制（0.5x/1x/2x）、背景音乐。H5 端 Phase 2 的动态播放应复刻这一叙事节奏，而不是另起一套表现规范。

## 2. 总体架构与数据归属

### 2.1 三方角色

```
┌──────────────┐     ┌──────────────────┐     ┌─────────────────┐
│  TravelPin   │     │  华为云空间        │     │  自建服务器      │
│  (HarmonyOS) │     │  (Cloud DB +      │     │  (Express +     │
│              │     │   Cloud Storage)  │     │   Nginx)        │
├──────────────┤     ├──────────────────┤     ├─────────────────┤
│ 用户旅行数据  │────>│ 永久存储 + 同步    │     │                 │
│ 本地组装分享  │     │                  │     │ 临时缓存分享数据  │
│ 调起系统分享  │     │ 原始照片归属      │     │ H5 模板渲染      │
│              │     │                  │     │ 签名 URL 校验     │
│              │─────────────────────────────>│ 限时链接生成      │
└──────────────┘     └──────────────────┘     └─────────────────┘
```

### 2.2 关键约束

- **华为云空间**是用户数据的永久归属，旅行数据与原始照片以云空间为准。
- **自建服务器**是无状态服务加临时缓存，不承担长期持久化职责。
- **分享链接必须限时有效**，过期后服务端拒绝访问并清理缓存。
- **分享分发不能只停留在裸 URL**，需要具备可视化卡片封装能力，以适配 QQ / 微信聊天分发场景。
- **HMAC 密钥仅保留在服务端**，App 端不参与签名。

### 2.3 端到端主链路

```
模块 1 App 前端收集与上传
  ↓
模块 2 服务端接收、存储、组装 H5、生成限时链接
  ↓
模块 3 App 前端封装限时链接、调起 QQ/微信分享、验证 QQ/微信浏览器打开
```

## 3. 模块一：前端收集打包路线的图文信息发送

### 3.1 模块目标

App 端负责从本地数据层读取路线叙事所需的最小数据集，完成图文打包、照片清洗、请求组装，并将分享请求发往自建服务器。

### 3.2 输入与输出

输入：

- `tripId`
- 路线元数据
- 节点列表
- 节点关联照片

输出：

- `multipart/form-data` 分享发布请求

### 3.3 前端处理流程

```
SharePage
  ├── 1. 从 RDB 加载 trip 与节点数据
  ├── 2. 读取节点对应照片文件
  ├── 3. 进行 EXIF 清洗，移除 GPS / 设备信息
  ├── 4. 组装 ShareTripRequest JSON
  ├── 5. 将照片按 nodeOrder 扁平化，并以 `photo_N` 连续编号写入 multipart
  ├── 6. 发送 POST /api/v1/share/publish
  └── 7. 等待服务端返回限时 URL、shortCode 与过期时间
```

### 3.4 `POST /api/v1/share/publish`

Content-Type: `multipart/form-data`

| 字段 | 类型 | 说明 |
|------|------|------|
| `tripData` | JSON (text) | 路线元数据与节点列表 |
| `photo_N` | File (binary) | 照片文件，N 从 0 开始连续编号，按节点顺序扁平排列 |
| `expiryHours` | number | 链接有效期，默认 168 小时 |

`tripData` 示例：

```json
{
  "tripId": "trip_abc123",
  "tripName": "深圳湾周末骑行",
  "totalDistance": 12.5,
  "coverIndex": 0,
  "createdAt": 1714200000000,
  "nodes": [
    {
      "id": "node_001",
      "title": "红树林公园",
      "content": "下午三点到达，阳光正好...",
      "poiName": "深圳湾红树林",
      "latitude": 22.5231,
      "longitude": 113.9432,
      "photoCount": 2,
      "mood": "惬意",
      "tags": ["骑行", "日落", "自然"],
      "visitedAt": 1714200000000,
      "nodeOrder": 0
    }
  ]
}
```

### 3.5 前端数据约束

- 仅发送 H5 渲染所需字段，不发送 `cloudId`、`syncStatus`、`ownerUid` 等内部字段。
- `photo_N` 上传前必须完成 EXIF 清洗，且 N 必须从 0 开始连续编号。
- 节点照片顺序必须可由 `photoCount + nodeOrder` 还原。
- 当节点和图片量过大时，前端应限制单次分享体量，避免上传超时。

### 3.6 模块一任务分配

| 任务 | 说明 | 责任建议 |
|------|------|------|
| 数据读取 | 从 RDB 装配 trip 与 nodes | App 前端 |
| 图片清洗 | 复用现有 EXIF 工具链 | App 前端 |
| 请求组装 | multipart 构建与错误处理 | App 前端 |
| 发布页 UI | 状态管理、上传态、失败重试 | App 前端 |

## 4. 模块二：后端服务器接收、存储数据并组装展示网页，同时生成限时链接发送回前端

### 4.1 模块目标

服务端负责接收 App 端上传的数据，进行安全处理与临时存储，生成供外部访问的 H5 页面，并返回限时签名 URL 给 App。

### 4.2 服务端处理流程

```
POST /api/v1/share/publish
  ├── 1. 解析 multipart，请求体拆分为 tripData + photo_N[]
  ├── 2. 按 photoCount 将扁平图片回填到节点
  ├── 3. 再次做服务端兜底校验
  ├── 4. 统一转码 WebP（375w / 750w）
  ├── 5. 生成 shortCode（8 位随机字符串，冲突重试）
  ├── 6. 生成 HMAC 签名与 expiryTimestamp
  ├── 7. 写入内存或 Redis 临时缓存
  ├── 8. 生成 H5 访问 URL
  └── 9. 返回给 App 端
```

响应示例：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "url": "https://tp.app/s/xK9mNz?t=1714800000&s=a7f2c9b3d1e4...",
    "shortCode": "xK9mNzA1",
    "expiresAt": 1714800000
  }
}
```

返回语义：

- HTTP 状态码为 `201 Created`
- `url` 用于分享与访问
- `shortCode` 用于调试、状态查询或缓存定位
- `expiresAt` 为秒级 Unix 时间戳

### 4.3 H5 页面访问接口

**`GET /s/:shortCode?t=<expiry>&s=<signature>`**

校验流程：

1. 读取 `shortCode`、`t`、`s`
2. 校验 `HMAC-SHA256(shortCode + ":" + t, SECRET) == s`
3. 校验当前时间未超过 `t`
4. 从缓存中读取 trip 数据与转码后的图片 URL
5. 将数据注入 EJS 模板并输出完整 HTML

### 4.4 H5 模板职责

服务端输出的 H5 页不只是“能打开”，还必须承担后续“卡片化分享预览”的元数据承载职责。页面至少需要包含：

- 标题：路线名 + 产品名
- 摘要：路线简介、节点数、总距离等
- 封面图：路线封面或首图
- 可供 QQ / 微信抓取的分享元信息
- Phase 1 静态图文内容
- Phase 2 地图回放动画容器

### 4.5 分享卡片所需元数据

为支持“不是裸 URL，而是直观、美观的卡片分享”，服务端返回的 H5 页面 head 中必须输出可被社交容器识别的元信息，例如：

- `title`
- `meta name="description"`
- `meta property="og:title"`
- `meta property="og:description"`
- `meta property="og:image"`
- `meta property="og:type"`
- `meta property="og:url"`

同时应预留微信生态所需的可扩展配置位，例如：

- JS-SDK 签名注入位
- 分享标题 / 摘要 / 缩略图的运行时覆盖配置
- 针对 QQ / 微信内置浏览器的兼容脚本入口

### 4.6 缓存与过期策略

- 缓存内容：trip 元数据、节点文本、WebP 图片映射、封面图、过期时间
- 存储介质：内存缓存或 Redis
- 过期策略：TTL 与 `expiryHours` 一致
- 清理方式：
  - Lazy delete：访问过期链接时即时删除
  - Cron：按小时扫描并批量清理

### 4.7 模块二任务分配

| 任务 | 说明 | 责任建议 |
|------|------|------|
| 发布接口 | multipart 解析、签名与响应 | 后端 |
| 媒体处理 | WebP 转码、封面抽取 | 后端 |
| 临时存储 | Redis / 内存缓存与 TTL | 后端 |
| H5 模板 | EJS 页面渲染与元信息输出 | 后端 / H5 |
| 访问校验 | HMAC、过期判定、异常页 | 后端 |

## 5. 模块三：前端可视化封装限时链接并支持 QQ、微信分享以及验证对应浏览器中网页的正常

### 5.1 模块目标

从服务端返回限时 URL 开始，前端需要将“可访问链接”进一步提升为“可分发的社交分享对象”，使其在系统分享面板、QQ、微信以及剪贴板分发链路中具备卡片化展示能力，并完成在 QQ / 微信浏览器中的打开验证。

职责边界补充：

- App 端在接口层面的核心职责仍然是：读取数据、发起 `publish` 请求、消费服务端返回的分享 URL。
- 分享预览卡片、QQ/微信分发、剪贴板回退属于拿到 URL 之后的前端交互层职责，不参与签名生成、H5 页面组装或服务端缓存管理。

### 5.2 模块边界

本模块从以下输入开始：

- 服务端返回的 `url`
- 服务端返回的 `expiresAt`
- 当前路线的标题、摘要、封面图、节点数量、距离等前端已有信息

本模块输出以下结果：

- App 内可展示的分享预览卡片
- 可被系统分享面板分发的分享对象
- 可通过剪贴板粘贴传播并在 QQ / 微信中转为卡片预览的 H5 链接
- 已验证能在 QQ 浏览器 / 微信浏览器中正常打开的落地页

### 5.3 关键设计原则

- **分享对象不能仅是裸 URL 文本**。即便底层依然是 URL，分发时也要带有标题、摘要、封面图、来源标识。
- **卡片感知分两层完成**：
  - App 内先生成一层本地可视化预览，给用户确认“将分享什么”
  - H5 页再提供一层社交抓取元信息，保证粘贴到 QQ / 微信后尽可能呈现卡片预览
- **系统分享与剪贴板并行存在**：
  - 优先走原生分享
  - 失败或目标应用不在分享面板中时，退化为复制链接 + 引导粘贴
- **验证不是可选项**。必须把“QQ / 微信内置浏览器打开是否正常”作为交付验收条件，而不是上线后再排查

### 5.4 从限时 URL 到可视化分享卡片的前端封装流程

```
服务端返回 url
  ├── 1. 前端解析并保存 url / expiresAt
  ├── 2. 生成 SharePreviewModel
  │     ├── title
  │     ├── summary
  │     ├── coverImage
  │     ├── expiryText
  │     └── targetUrl
  ├── 3. 在 SharePage 上展示本地预览卡片
  ├── 4. 构建 SharePayload
  │     ├── 分享标题
  │     ├── 分享摘要
  │     ├── 缩略图/封面引用
  │     └── 限时 H5 URL
  ├── 5. 调起系统分享面板
  └── 6. 若系统直达失败，则回退到剪贴板链路
```

### 5.5 前端本地预览卡片要求

SharePage 在拿到限时 URL 后，不应只显示“复制链接成功”或一行 URL，而应显示一个明确的分享预览区，至少包含：

- 路线标题
- 封面图
- 路线摘要（距离 / 节点数 / 时间）
- “分享给微信 / 分享给 QQ / 复制链接”操作入口

该预览卡片的作用有两点：

1. 用户确认分享内容是否正确
2. 为系统分享提供一致的标题、摘要、封面来源

### 5.6 系统分享链路

目标是优先通过系统分享面板将分享对象交给本机 QQ 或微信，而不是默认发出一段纯文本。

推荐分享载荷结构：

- `title`: 如“深圳湾周末骑行 - TravelPin”
- `description`: 如“4 个节点 · 12.5km · 7 天内有效”
- `url`: 服务端返回的限时链接
- `thumbnail`: 本地封面图或服务端封面图引用

系统分享行为定义：

1. 前端构造统一 `SharePayload`
2. 通过 `systemShare` 调起系统分享面板
3. 优先让用户选择 QQ 或微信
4. 若目标应用支持 URL 分享与卡片抓取，则由目标应用基于页面元信息展示卡片
5. 若分享面板未出现目标应用、或仅能发送纯文本，则自动降级到剪贴板方案

### 5.7 剪贴板分发链路

剪贴板不是简单兜底，而是必须设计成可用的第二主链路。

处理要求：

1. 复制内容优先为**限时 URL 本身**
2. App 内同时提示“已复制，可粘贴到微信 / QQ 对话框发送”
3. 服务端 H5 页面必须具备完整社交元信息，以便在用户粘贴 URL 后，由 QQ / 微信侧抓取并转成预览卡片
4. 若目标聊天场景只显示文本链接，也仍需保证点开后页面完整可用

也就是说，**“卡片化”并不要求前端把链接变成图片再发送**，而是要求前端和 H5 页面共同提供足够的封装信息，使分享目标应用在 URL 分发时能生成视觉化预览。

### 5.8 微信 / QQ 卡片化落地要求

为了让微信和 QQ 更稳定地把限时 URL 呈现为卡片而不是裸链，需满足以下条件：

- 页面首屏可被抓取时获得明确标题、摘要、封面图
- 封面图 URL 可公网访问，尺寸与体积适合预览抓取
- 页面返回状态码稳定，不能依赖登录态
- 页面 head 中输出完整分享元标签
- 页面在首次打开时首屏 HTML 就有核心信息，不能完全依赖前端二次请求后再渲染
- 针对微信场景预留 JS-SDK 分享配置能力，用于在微信内二次分享时保持卡片信息一致

### 5.9 QQ / 微信浏览器打开验证链路

本模块必须包含验证闭环，直到能确认“在 QQ 或微信浏览器下打开 URL 网页正常”为止。

验证对象：

- 微信聊天中打开链接后的微信内置浏览器
- QQ 聊天中打开链接后的 QQ / X5 内置浏览器

验证步骤：

1. 从 App 成功获得限时 URL
2. 通过系统分享面板分享到微信或 QQ
3. 或通过剪贴板粘贴到微信 / QQ 对话框发送
4. 在目标应用中确认消息形态：
   - 是否出现分享卡片
   - 若未出现，是否至少为可点击链接
5. 点击链接进入对应内置浏览器
6. 验证 H5 页面：
   - 是否成功打开，无 403 / 404 / 空白页
   - 标题、封面、正文是否正常
   - 图片是否正常加载
   - Phase 1 静态内容是否完整
   - Phase 2 动画在支持环境下是否可运行
   - 微信环境下音频/交互是否按兼容策略降级
   - 倒计时与过期态是否正确
7. 对过期链接重复验证：
   - 是否显示明确过期提示页
   - 是否不再泄露原始 trip 数据

### 5.10 模块三验收标准

满足以下条件才能视为完成：

- App 内存在清晰的分享预览卡片，而非只展示 URL
- 系统分享面板可以发起到 QQ / 微信的分享尝试
- 剪贴板链路可正常工作
- 粘贴到 QQ / 微信后，目标应用能够尽可能渲染分享卡片；即使退化为文本链接，也能正常打开
- 在微信浏览器与 QQ 浏览器中打开限时链接时，H5 页面渲染正常
- 过期链接在两个环境下都表现为统一的失效页

### 5.11 模块三任务分配

| 任务 | 说明 | 责任建议 |
|------|------|------|
| 预览卡片 UI | 本地可视化分享预览 | App 前端 |
| 分享载荷封装 | title / description / thumbnail / url | App 前端 |
| 系统分享接入 | 调起系统面板并优先投递 QQ / 微信 | App 前端 |
| 剪贴板回退 | 复制、提示、引导粘贴 | App 前端 |
| 元信息适配 | 页面 meta / OG / 微信配置 | H5 / 后端 |
| 联调验证 | QQ / 微信消息形态与浏览器打开测试 | App 前端 + H5 + QA |

interface ShareRouteData {
  tripId: string;

  content: string;

  url: string;
  width: number;

## 7. Phase 1：H5 静态展示 (MVP)

### 7.1 技术栈

| 技术 | 选型 | 理由 |
|------|------|------|
| 构建工具 | Vite | 轻量、快速 |
| 页面模板 | HTML5 + CSS3 + JS | 独立于 HarmonyOS |
| 服务端渲染 | Node.js (Express) + EJS | 动态注入数据与分享元信息 |
| 静态资源托管 | Nginx | 基础设施已具备 |
| 微信兼容 | `WeixinJSBridgeReady` | 微信内浏览器能力接入 |

### 7.2 服务端注入模板示例

```html
<!DOCTYPE html>
<html>
<head>
  <meta name="viewport" content="width=device-width,initial-scale=1">
  <title><%= tripName %> - TravelPin</title>
  <meta name="description" content="<%= shareDescription %>">
  <meta property="og:title" content="<%= tripName %> - TravelPin">
  <meta property="og:description" content="<%= shareDescription %>">
  <meta property="og:image" content="<%= coverPhotoUrl %>">
  <meta property="og:url" content="<%= shareUrl %>">
  <link rel="stylesheet" href="/assets/share-page.css">
</head>
<body>
  <script>
    window.__ROUTE_DATA__ = <%- JSON.stringify(routeData) %>;
  </script>
  <div id="app"></div>
  <script type="module" src="/assets/share-page.js"></script>
</body>
</html>
```

### 7.3 页面职责

- 首屏即能展示封面、标题、摘要
- 节点图文按 `nodeOrder` 展示
- 显示剩余有效时间
- 为微信 / QQ 抓取提供稳定首屏内容

## 8. Phase 2：H5 动态路线回放

### 8.1 动画标准

对齐 `TripReplayPage`：

```
FADE_IN 2.0s → STAYING 3.0s → FADE_OUT 1.0s → MOVING 1.5s
```

### 8.2 推荐技术组合

| 方案 | 结论 |
|------|------|
| PixiJS + GSAP | 推荐 |
| Canvas 2D | 可作为降级 |
| CSS Transform + DOM | 不推荐 |
| MapLibre GL JS | 仅作备选 |

### 8.3 H5 端状态机

```typescript
enum PlayPhase {
  IDLE = 0,
  FADE_IN = 1,
  STAYING = 2,
  FADE_OUT = 3,
  MOVING = 4
}

interface PlaybackState {
  currentPhase: PlayPhase;
  currentNodeIndex: number;
  speed: 0.5 | 1 | 2;
  isPlaying: boolean;
  progress: number;
}
```

## 11. 风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| 系统分享面板中微信 / QQ 不出现 | 无法直接分享 | 保留剪贴板链路，提供引导文案 |
| 粘贴后未渲染卡片 | 分享视觉效果不足 | 完善 H5 meta 与封面抓取能力 |
| 微信 / QQ 内置浏览器兼容问题 | 白屏、音频异常、动画失效 | 做环境检测与降级策略 |
| 链接过期策略被缓存干扰 | 用户打开异常 | 服务端始终以签名与时间戳校验为准 |
| 大体积上传超时 | 分享失败 | 控制图片数量与尺寸，支持重试 |
| 服务端缓存膨胀 | 资源占满 | 严格 TTL 与定时清理 |
