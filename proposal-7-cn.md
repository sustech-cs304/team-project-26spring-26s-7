# 项目提案：鸿蒙旅行日记应用

## Part I. 初步需求分析

### 项目概述

**TravelPin** 是一款基于地理位置的的旅行日记应用。

用户可以将照片与配文精准钉在地图坐标上，一键生成动态旅行剪辑，并借助 AI 智能生成不同风格文案分享至社交平台。
系统采用本地前端、服务器后端与远程云端的三层架构，主打“离线优先”，在无网环境下也能流畅记录。同时，通过强制元数据剥离和加密签名分享，系统全方位保障用户隐私与数据安全。

### 1. 功能需求

本系统包含以下 5 项正交的功能特性：

**交互式地图：** 
* 将照片、文字故事和心情标签以"记忆节点"的形式钉在真实地图上。
* 按时间范围、地理区域或自定义标签浏览和筛选节点。
* 节点密集时杜绝卡顿，支持聚合渲染，点击聚合簇可展开查看各条目详情。

**动态旅程回放：**
* 选择并组合记忆节点为一条旅行线路。
* 一键生成充满回忆的旅行剪辑。

**跨平台路线分享：** 
* 为任意路线生成加密签名、限时有效的 URL。
* 支持直接分享至微信或复制到系统剪贴板。
* 接收者无需安装 App 即可查看完整旅程。

**AI 辅助生成文案：**
* 用户可以输入简单草稿，系统结合旅途元信息生成社交媒体文案。
* 支持用户选择语气风格（如诗意、简洁、幽默）。

**多设备无缝同步：**
* 支持华为账号一键授权登录 
* 记忆节点、路线和草稿在手机、平板、电脑之间同步。
* 在手机随手记录，在电脑大屏上继续编辑。

### 2. 非功能需求

#### 2.1 易用性
* **交互效率：** 核心操作（添加/删除/修改节点、创建路线）应在主页面少数次点击内完成。
* **学习成本：** 系统界面设计与交互操作需保持简洁，帮助用户快速上手。
* **状态反馈：** 界面需对所有后台行为提供明确且非阻塞的视觉或触觉反馈，避免用户对系统状态产生困惑。
* **无障碍与适配：** 系统支持动态字体缩放，以服务不同年龄段的用户。

#### 2.2 性能
* **地图渲染：** 动态聚合并选择性渲染节点与路线，确保在多数据点场景下也能保持流畅渲染。
* **AI 响应时间：** LLM 文案请求应在较短时间内输出首个 token（TTFB），并采取流式传输以保持交互的即时感。
* **应用启动：** 在标准 HarmonyOS 设备上，冷启动到可交互地图页面应在极短时间内完成。

#### 2.3 安全与隐私
* **强制元数据剥离：** 所有隐私数据（如设备型号、坐标、时间戳）在任何同步或分享操作之前，必须于本地层强制剥离。
* **安全分享：** 分享链接必须使用 HMAC-SHA256 签名并附带 TTL 过期时间；禁止使用可猜测的顺序 ID。
* **AI 内容合规：** 所有 AI 生成的文案在展示给用户或进行持久化存储之前，必须通过内容审核过滤器。

#### 2.4 可靠性与可用性
* **离线优先：** 在完全无网络环境下，应用仍可正常使用基础功能（如创建节点、编辑草稿、浏览历史）；系统需以本地存储为主要数据源。
* **数据恢复：** 提供容错机制，用户删除数据后在设定的一定时间内可以进行数据恢复。

### 3. 技术需求

系统的运行环境、开发工具链和运行时技术栈如下：

* **客户端/前端（HarmonyOS App）：**
    * **语言与 UI 框架：** ArkTS 搭配 ArkUI 声明式 UI 框架，实现原生组件渲染。
    * **定位服务：** HarmonyOS `LocationHub` API，用于高精度、低功耗的 GPS 坐标获取和持续追踪。
    * **媒体访问：** HarmonyOS 系统级 Photo Picker（最小权限模型——无需获取相册完整访问权限）。
    * **本地持久化：** HarmonyOS 关系型数据库（RDB）存储结构化数据；本地文件系统存储媒体和缓存。
* **后端与云端（分布式服务）：**
    * **同步服务器：** 基于版本化更新队列实现最终一致性的分布式同步服务，用于多设备冲突解决。
    * **空间数据库：** sqllite 用于大规模地理数据的高效空间索引、聚合查询和地理围栏。
    * **对象存储：** 云端 OSS（对象存储服务），用于集中托管媒体文件并通过 CDN 分发缩略图。
    * **网页分享门户：** 轻量级只读 Web 应用，用于渲染分享的旅行路线，可从任何浏览器访问。
* **AI 与计算服务：**
    * **本地边缘 AI：** 设备端轻量级 ML 模型，用于 OCR 和文本向量化——支持本地搜索和索引，同时保护隐私。
    * **云端 LLM：** 通过 AI 网关访问的远程大语言模型，用于文本到文本的文案生成（从旅程元数据到风格化文案）和内容合规审核。
* **开发与交付环境：**
    * **IDE：** DevEco Studio（HarmonyOS 官方 IDE），用于前端开发。
    * **版本控制：** Git + GitHub；基于分支的工作流，配合 Pull Request 代码审查。
    * **项目管理：** GitHub Projects 看板，用于 Sprint 规划和进度追踪。
    * **CI/CD：** GitHub Actions，用于每次 PR 的自动化代码检查、测试和构建验证。
    * **测试：** 核心逻辑模块的单元测试；同步和 API 层的集成测试。

### 4. 数据需求

* **需要哪些数据：**
    * **地理空间数据：** 实时 GPS 坐标（经度、纬度）、移动轨迹点、海拔、航向和时间戳——构成所有地图和路线功能的基础。
    * **媒体与内容数据：** 用户选择的照片和视频（通过 Photo Picker 访问）、手动输入的文字草稿、心情标签和旅行元数据（名称、日期范围、封面图片）。
    * **AI 上下文数据：** 从记忆节点聚合的旅程元数据——途经 POI 名称（通过逆地理编码获取）、总距离、旅行时长和用户提供的草稿文本——用作文案生成的输入。不向云端 LLM 发送任何照片。
    * **用户与同步数据：** 用户画像、设备标识符、同步版本向量和冲突解决日志。
* **如何获取数据：**
    * **位置：** 在用户明确授权后，通过 HarmonyOS `LocationHub` API 独占获取；持续追踪仅在旅行记录会话期间激活。
    * **媒体：** 通过系统级 Photo Picker 访问，遵循最小权限原则——应用永远不会请求相册完整访问权限。
    * **POI 与地理编码：** 通过地图服务 API 进行逆地理编码，将坐标解析为可读的地点名称。
    * **AI 输入：** 旅程元数据（POI 列表、距离、时长）在本地聚合后，作为结构化文本发送至云端 LLM 网关；用户草稿文本为可选的附加输入。
* **数据处理与存储策略：**
    * **隐私清洗：** 所有媒体在离开设备之前，于本地强制执行 EXIF 元数据剥离。
    * **冷热分层：** 活跃旅行数据以高频率写入本地 RDB（热数据）；历史和归档旅行在网络空闲时异步推送至云端 PostGIS 和 OSS（冷数据）。
    * **数据保留：** 用户可永久删除任何节点或旅行，触发从本地存储、云端数据库和对象存储的级联删除。分享链接数据在 TTL 到期后自动失效。
