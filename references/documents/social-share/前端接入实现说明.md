## 2026-05-05 · 迭代 v0.5.2 · QQ 系统分享出卡片预览（接 thumbnail）

### 背景

v0.5.1 修了"微信不出现在分享面板"，但 QQ 仍只显示纯链接、不出卡片。原因：QQ 的接收方逻辑**不会主动 fetch URL** 抓 OG meta（不像微信），它需要分享方在 SharedData 里直接放 `thumbnail: ArrayBuffer` 才能渲染卡片。

### 改动

| 文件 | 类型 | 说明 |
|---|---|---|
| [`feature/social-share/types/ShareTypes.ets`](../../frontend/entry/src/main/ets/feature/social-share/types/ShareTypes.ets) | 修改 | `SharePublishOk` 加可选字段 `coverPhotoUrl?: string`（后端 v0.7.1 起返回） |
| [`feature/social-share/services/SystemShareService.ets`](../../frontend/entry/src/main/ets/feature/social-share/services/SystemShareService.ets) | 修改 | `SystemSharePayload` 加可选 `thumbnail?: ArrayBuffer`，传到 `SharedData` 构造里 |
| [`feature/social-share/pages/SharePage.ets`](../../frontend/entry/src/main/ets/feature/social-share/pages/SharePage.ets) | 修改 | 1) 加 private `coverPhotoUrl: string`，publish 成功时记下；2) 加 `fetchCoverThumbnail()` 用 `@kit.NetworkKit.http` GET 成 ArrayBuffer，3s 超时；3) `handleShare()` 调它 → 塞进 payload；4) 任何错误返 undefined 不阻塞分享 |

### 数据流

```
publish 响应 (v0.7.1 起)
  data.coverPhotoUrl = "https://.../cache/k7OlDyDS/0_375w.webp"
       ↓ 存到 SharePage.coverPhotoUrl
       ↓
用户点 "QQ" / "微信"
  ↓
SharePage.handleShare
  ↓
fetchCoverThumbnail() — http GET 上面那个 URL，3s 超时
  ↓ ArrayBuffer (≈30 KB WebP)
  ↓
SystemShareService.share({ url, title, description, thumbnail })
  ↓ thumbnail 透传到 SharedData
  ↓
ShareKit 系统分享面板
  ↓ 用户选 QQ
  ↓
QQ 接收方 → 渲染卡片（标题 + 描述 + thumbnail）
```

### 容错链

每一步失败都不让分享按钮卡住：

| 失败 | 行为 |
|---|---|
| 后端没返 coverPhotoUrl（0 照片 trip） | `coverPhotoUrl=''` → fetchCoverThumbnail 直接返 undefined → share 不带 thumbnail（接收方仍按 URL 文本渲染） |
| http GET 超时 / 5xx / 网络断 | catch 返 undefined，同上 |
| systemShare 调用挂掉 | 走原 v0.5 兜底（剪贴板 + toast"已复制内容"） |

### UI 影响

无 — 用户看不到任何变化。只是点 QQ 的时候底层多了 0–500ms 的 fetch 延迟。WebP 图小（30 KB 量级），通常感知不到。

### 真机验收

1. 进 SharePage，等链接生成
2. 点 **QQ**
3. 应该看到系统分享面板，选"QQ" → 进 QQ 联系人选择 → 选会话 → 会话里出现的应该是**一张带封面图 + 标题 + 摘要的卡片**（不是裸链）
4. 微信路径同样能看卡片（可以选 thumbnail 路径或 OG fetch 路径，先到先得）

### 仍未做

1. **预加载 thumbnail**：当前是用户点击时才 fetch，加了 0–500ms 延迟。可以改成 publish 成功后立刻后台预热，下次再做
2. **缩略图缓存**：每次切档重新 publish 都要重新 fetch 同一张（理论上 backend 也重新生成，但实际上对同一份 trip 内容是相同的）。优化空间但不紧迫
3. **App 内本地预览卡片** UI（v4 §5.5 要求）—— 仍未做
4. **倒计时 / QR** —— 仍未做

## 2026-05-05 · 迭代 v0.5.1 · 修系统分享面板兼容性 + 平台调整

### 背景

v0.5 上线后真机测试发现**系统分享面板里没有微信**。两个修复：

### 1. 把 UTD 从 `general.url` 改成 `general.hyperlink`

`general.url` 在 HMOS 标准 Uniform Data Type 体系里是"裸 URL 字符串"语义；`general.hyperlink` 是"URL + 描述文字"，对应分享卡片这个使用场景。**多数 app 注册的是 hyperlink 类型**（QQ / 微博 / 邮件 / 浏览器都至少包含这个），而早期 HMOS 微信原生版可能只声明了 hyperlink，没声明 url。

同时再附加一份 `general.text` record（用 `data.addRecord()`），让那些只声明文本接收的 app 也能看到这次分享（双倍兜底）。

[`SystemShareService.share`](../../frontend/entry/src/main/ets/feature/social-share/services/SystemShareService.ets) 同步修改 + 加详尽日志（含 BusinessError code）方便真机排查。

### 2. 「微博」按钮改成「QQ」

[`SharePage.sharePlatforms`](../../frontend/entry/src/main/ets/feature/social-share/pages/SharePage.ets):

```ts
private sharePlatforms: SharePlatform[] = [
  new SharePlatform('W', '微信'),
  new SharePlatform('Q', 'QQ'),       // ← 原 ('B', '微博')
  new SharePlatform('C', '复制链接'),
]
```

按钮 `handleShare(platform)` 的逻辑不区分 platform 名（除了"复制链接"），都调 `SystemShareService.share`。所以这个改动只是 UI 文案，没有逻辑分支变化。

### 微信仍不出现的诊断步骤

如果改了 UTD 后微信还是不在面板里出现：

1. **真机微信版本是不是 HMOS 原生版**？
   - 设置 → 应用 → 微信 → 版本信息，看是不是 Native HarmonyOS（不是 Android 兼容运行的那种）
   - Android 兼容版微信不会出现在 HMOS 系统分享面板里
2. **从其它系统 app 测一次**：相册选一张图 → 系统"分享"按钮 → 微信会不会出现？
   - 出现 → 我们的 SystemShareService 调用方式有问题
   - 不出现 → 设备上的微信本身没注册系统分享（卸载重装 / 等微信新版）
3. **看 DevEco 日志**，TAG 过滤 `SystemShareService`：
   ```
   I SystemShareService: share() url=... title="..."
   I SystemShareService: systemShare panel shown successfully
   ```
   如果只看到 `share() url=...` 但没有 `panel shown successfully`，说明 `controller.show` 抛了异常 —— 看 error 行的 `code=...` 提示
4. **试切 UTD**：把 `general.hyperlink` 改成 `general.text`，再测一次。如果切成 text 微信就出现了，说明微信声明的 UTD 不含 hyperlink，需要再调

### 完全没办法用 systemShare 怎么办？

如果上述都试过微信还是不出现，最后兜底是 **WeChat URL Schemes** 方案：用 `Want` + `startAbility` 通过 `weixin://` 协议直接拉起微信，把链接 + 描述塞进去。这绕过 systemShare，用 deep link 做到等价效果。但需要硬编码 `bundleName: 'com.tencent.mm'`，对其它平台（QQ 等）就要再写一套。下迭代再做。

## 2026-05-05 · 迭代 v0.5 · 接入 systemShare（搭配后端 v0.7 OG meta）

### 目的

v0.4 的"微信 / 微博"按钮还是退化到剪贴板复制。v0.5 把它换成真正的 `@kit.ShareKit.systemShare`，调起系统分享面板让 OS 把 URL + 标题 + 摘要投递给目标 app。配合后端 v0.7 在 H5 viewer head 输出 OG / Twitter / wxcard 三套 meta，多数环境下都能在 QQ/微信/微博中渲染成卡片预览。

### 改动一览

| [`feature/social-share/pages/SharePage.ets`](../../frontend/entry/src/main/ets/feature/social-share/pages/SharePage.ets) | 修改 | `handleShare(platform)` 微信/微博分支换成调 `SystemShareService.share()`；返 false 时退回原剪贴板路径并显示"系统分享不可用，已复制内容…" |
| [`feature/social-share/index.ets`](../../frontend/entry/src/main/ets/feature/social-share/index.ets) | 修改 | 导出 `SystemShareService` / `SystemSharePayload` |

### SharePayload 内容

```ts
{
  url:         this.shareUrl,                              // 后端签名链接
  title:       `${this.currentTripName} - TravelPin`,      // 卡片标题
  description: this.copyText,                              // 用户编辑的分享文案
}
```

ShareKit 用 `utd: 'general.url'`（Uniform Data Type 标准 hyperlink 描述符），让接收方应用按"链接 + 元信息"理解，并自行抓 URL 的 OG meta 渲染卡片。

### 兼容性

- ShareKit 在 **HarmonyOS NEXT API 12+** 才有；项目其它 feature（MapKit 等）也用 NEXT 才有的 kit，所以这条要求和现有项目对齐
- 调用方任何失败都会被 SystemShareService 吞下并 log 一条 warn，**不会让 SharePage 崩**；用户看到"系统分享不可用，已复制内容…"toast，能正常完成分享路径

| 点 "微博" | 同上 | 同上 |
| 点 "复制链接" | 复制 URL → toast "链接已复制" | **未改** |
| ShareKit 不可用 | — | 自动回退到 v0.4 剪贴板路径 |

## 2026-05-05 · 迭代 v0.4 · 切档自动撤销旧链接（防冗余）

### 背景

v0.3 让用户能切换 4 档过期时长，但每切一次 = 多一条活链接。用户不小心 / 恶意切多次都会让后端积累冗余分享。本迭代和后端 v0.6 同步实现"切档时自动撤销旧的"。

### 改动一览

| 文件 | 类型 | 说明 |
|---|---|---|
| [`feature/social-share/types/ShareTypes.ets`](../../frontend/entry/src/main/ets/feature/social-share/types/ShareTypes.ets) | 修改 | `SharePublishOk` 加 `sig: string`；新增 `ReplaceLink` 接口 |
| [`feature/social-share/services/ShareService.ets`](../../frontend/entry/src/main/ets/feature/social-share/services/ShareService.ets) | 修改 | `publish()` 加可选第 5 参数 `replace?: ReplaceLink`，写入三个 multipart 字段 |
| [`feature/social-share/pages/SharePage.ets`](../../frontend/entry/src/main/ets/feature/social-share/pages/SharePage.ets) | 修改 | 维护 `private lastLink: ReplaceLink \| null`；切档时把它作为 `replace` 传过去；toast/UI 文案区分"首次生成" vs "替换" |
| [`feature/social-share/index.ets`](../../frontend/entry/src/main/ets/feature/social-share/index.ets) | 修改 | 导出 `ReplaceLink` |

### UI 行为

| 状态 | 提示 / 按钮 |
|---|---|
| 进入页面，正在生成 | "正在生成签名链接…" |
| 首次生成成功 | toast: "分享链接已生成（有效期：XXX）" |
| 已生成 → 切换其它档 | 选项行下方多一行小字 "切换其它有效期会自动撤销当前链接" |
| 切档完成 | toast: "已替换为新链接（原链接已失效，有效期：XXX）" |

### 替换三件套从哪儿来

后端 v0.6 起在 `PublishOk` 直接返回 `sig` 字段（不用再解析 URL），所以每次成功后简单存：

```ts
this.lastLink = {
  shortCode: result.data.shortCode,
  expirySec: result.data.expiresAt,
  sig: result.data.sig,
}
```

下次 publish 把这个对象作为 `replace` 参数传给 `shareService.publish()` 即可。

### 不动的部分

- `cachedReq + cachedPhotos` 缓存机制不变 —— 切档**只重传照片 + 重转码**，不重读 RDB
- 5 分钟测试档不动 —— 仍是首选验证 5min 全生命周期 +"切档替换"两件事的好选项

### 验证清单（真机或 testclient 都行）

- [ ] 进入 SharePage，等首链接（默认 7 天）出来
- [ ] 复制 URL_A，浏览器打开能看到 H5 → ✅
- [ ] 选 "5 分钟（测试）" → 等新链接（≈3-15 秒）出来
- [ ] toast 应是 "已替换为新链接（原链接已失效…）"
- [ ] 浏览器刷新 URL_A → 应该 404
- [ ] 复制 URL_B，浏览器能看到 → ✅
- [ ] 5 分钟后刷新 URL_B → 410 → 再刷 404（lazy delete）
- [ ] 任意一档点已选中那个 → 不应有任何变化（前端守卫直接 return）

### 已知限制

1. **切档要重传照片**：因为后端是把 publish 当独立快照处理。同 trip 在 5 张原图情况下大约 5–15 秒。可接受。如果以后想优化，可以加 "extend-only" 端点（仅改 expiry 不重转码）
2. **没倒计时**：5 分钟档剩余多少时间不可见。下迭代加
3. **systemShare / QR / 预览卡片** 仍未做（v4 文档 §5 模块三验收必须项）

## 2026-05-04 · 迭代 v0.3 · 用户可选过期时长 + 链接生命周期

### 目的

### 改动总览

| 文件 | 类型 | 说明 |
|---|---|---|
| [`feature/social-share/services/ShareService.ets`](../../frontend/entry/src/main/ets/feature/social-share/services/ShareService.ets) | 修改 | `publish()` 加可选第 4 参数 `expiryMinutes`；传入且 > 0 时优先于 `expiryHours` 写到 multipart |
| [`feature/social-share/pages/SharePage.ets`](../../frontend/entry/src/main/ets/feature/social-share/pages/SharePage.ets) | 修改 | 4 档选择器 UI + 切换即重传 + trip/photos 缓存复用 + ExpiryOption 类 |

### 4 档选项

```ts
private expiryOptions: ExpiryOption[] = [
  new ExpiryOption('5 分钟（测试）', 0, 5),    // → expiryMinutes=5
  new ExpiryOption('1 天',          24, 0),    // → expiryHours=24
  new ExpiryOption('7 天',         168, 0),    // → expiryHours=168 (默认)
  new ExpiryOption('30 天',        720, 0),    // → expiryHours=720
]
```

`ExpiryOption.minutes > 0` 用 `expiryMinutes`，否则用 `expiryHours`。后端规则一致：`expiryMinutes` 优先于 `expiryHours`。

### 切档行为

```
用户进入 SharePage
   ↓
首次 publishShare（默认 7 天）
   ├ resolveTrip → 缓存 cachedReq + cachedPhotos
   └ POST /publish expiryHours=168 → 显示 URL
   ↓
用户点 "5 分钟（测试）"
   ↓
onPickExpiry(0)：selectedExpiryIdx = 0
   ↓
publishShare()  ← 复用 cachedReq + cachedPhotos
   └ POST /publish expiryMinutes=5 → 显示新 URL
```

注意：**切档会重新 multipart 上传** —— 5 张原图大约 5–15 秒，期间整个选择器置 `opacity: 0.5` 不可点。这次没做"延长链接"端点，等以后真有需求再加。

### 联调建议路径

1. DevEco Studio 把项目跑起来到真机或模拟器
2. 进 SharePage（不带 tripId 也行，自动取 RDB 最新一条）
3. 等首次链接出来 → 点 **"5 分钟（测试）"** 档
4. 复制新链接，浏览器打开看效果
5. **等 5 分钟**（喝杯水的功夫），刷新浏览器
6. 应该看到："此链接已过期"页面（HTTP 410）

3. **没有"链接快到期"提示**：下迭代加倒计时
4. **systemShare / QR / 预览卡片** 仍未做（按 v4 文档 §5 模块三验收必须项；下迭代正式接）

## 2026-04-30 · 迭代 v0.2 · 接通 RDB 真实数据

### 目的

v0.1 的 `SharePage` 还是用打包资源做 demo；v0.2 把 trip 元数据和照片**真的**从本机 RDB 取出来发给后端，做到一打开页面就能分享当前 / 指定 trip。

### 关键改动

只动了 [`feature/social-share/pages/SharePage.ets`](../../frontend/entry/src/main/ets/feature/social-share/pages/SharePage.ets) 一个文件；service / types 层不动，证明 v0.1 的接口分层是合理的。

新增内部方法（按调用顺序）：

| 方法 | 作用 |
|---|---|
| `resolveTrip()` | 三种结局：路由 tripId 命中 → 用之；未命中 → 报错；没传 tripId → 取 `getAllTravels()` 里 `createdAt` 最新的；RDB 完全空 → 落到 demo |
| `loadAndSortNodes(tripId)` | `RdbDataService.getNodesByTravelId()` + 按 `nodeOrder` 升序 |
| `buildRequestFromRdb(trip, nodes)` | 把 `Trip` + `MemoryNode[]` 转成契约 §4.2 的 `SharePublishRequest` |
| `collectPhotosAlignCount(req, nodes)` | 关键步骤：`SharePhotoHelper.toSharePhotos()` 会过滤掉**云端引用**和**缺失文件**，所以幸存数 ≤ `node.photos.length`。这里按幸存数**回写** `req.nodes[i].photoCount`，确保 `sum(photoCount) === photos.length`，否则后端返 `INVALID_PHOTO_COUNT (40001)` |

### resolveTrip 决策树

```
进入 SharePage
       │
       ▼
路由参数 tripId 存在？
   ├─ 是 ──→ getTravelById(tripId)
   │           ├─ 命中 ──→ 用真数据
   │           └─ 未命中 → 显示 "路线 X 不存在"
   │
   └─ 否 ──→ getAllTravels() 取 createdAt 最大的
              ├─ 列表非空 ──→ 用最新 trip
              └─ 列表为空 ──→ demo（保留作 fallback）
```

这意味着任何已经登录、有过至少一次旅行记录的用户，**直接打开 SharePage 就能立刻拿到自己最新一条 trip 的真实分享链接**，不需要再传参。

### 对 photoCount 的关键约束

后端按规范 §4.3 的"flat 列表 + 按 nodeOrder 切片"逻辑分配照片到节点。如果前端报的 `photoCount` 之和 ≠ 实际上传 `photo_N` 数，后端 `INVALID_PHOTO_COUNT (40001)`。

`MemoryNode.photos[]` 里可能混着：
- 沙箱 `file:///data/.../photos/photo_xxx.jpg` ← 可上传
- 云端引用 `users/{uid}/...` ← 跳过
- 已删除 / 未同步下来 → `fs.accessSync` 失败 ← 跳过

所以 `collectPhotosAlignCount` 必须按**幸存**数重写 photoCount，不能直接用 `node.photos.length`。

### 已知限制

1. **没做"用户选择哪个 trip 来分享"的 UI**：当前逻辑是"路由有 tripId 用之，否则取最新"。如果分享其它 trip，调用方在 `router.pushUrl` 时把 `tripId` 传进来即可。
2. **首次没有任何 trip 的全新用户会看到 demo trip**：刻意设计，便于演示和测试；正式上线时这条分支可改成"显示『还没有可分享的旅行』"。
3. **照片大小未在前端做压缩**：HarmonyOS 相册原图可能 4–12 MB / 张，后端单张限 20 MB / 总体 100 MB。一次分享 ≤ 5 张原图通常没问题，但建议下迭代加客户端 resize。

let cursor = 0

  const survived = SharePhotoHelper.toSharePhotos(nodes[i].photos)

### 与契约的对齐细节

| 契约条款 | 前端实现 |
|---|---|
| §4.1 multipart `tripData` text | `MultipartFormData.addField('tripData', JSON.stringify(trip))` |
| §4.1 multipart `photo_N` file | `MultipartFormData.addFile('photo_${i}', filename, path, contentType)` |
| §4.1 `expiryHours` text 默认 168 | `SharePage.expireDays * 24`，默认 `expireDays=7` |
| §4.2 `tripData.nodes[].photoCount` | 客户端先校验 sum；不匹配直接拒绝（不发请求） |
| §4.4 `code: 0` 成功 | `ShareService` 在 envelope 里检测 `code === 0 && data` 才视为成功 |
| §4.5 各错误码透传 | `ShareCallResult.errorCode + errorMessage + errorDetail` |
| §11 类型签名 | 一字不差，外加可选 `latitude/longitude`（对齐版文档加的字段） |
