# 技术需求文档：地图叙事 H5 分享系统 (项目对齐版 v3)

> **文档状态**: 基于 v2 修订，新增华为云空间角色与图片直传方案。
> **对齐日期**: 2026-04-28
> **目标分支**: `feature/social-share`

---

## 1. 项目概览

本项目为 TravelPin（鸿蒙旅行日记 App）的社交分享模块，核心目标：

- **App 端**：收集路线数据 → 连同照片直传自建服务器 → 获取签名分享链接 → 通过系统分享面板或剪贴板分发。
- **H5 端**：接收者在浏览器中打开链接后，看到限时有效的 H5 页面——Phase 1 为静态图文排版，Phase 2 升级为带地图动画的叙事回放。

**不包含的范围**：海报图片（PNG）的生成。本期仅实现 H5 页面分享。

**与现有 replay 功能的关系**：

> TripReplayPage (`feature/map-travel/pages/TripReplayPage.ets`) 已实现完整的路线回放动画系统——5 阶段状态机（淡入 2s → 停留 3s → 淡出 1s → 相机移动 1.5s）、bearing/tilt 3D 视角、离散进度条、速度控制（0.5x/1x/2x）、背景音乐。H5 端 Phase 2 的动态动画应**复刻这一叙事节奏**，而非重新设计。


**关键设计决策**：

| 决策 | 说明 |
|------|------|
| **照片由 App 端直发** | 复用 AI 文案 feature 已有通道（App → HttpClient → multipart → 自建服务器）；服务端不从华为云 Storage 下载，避免 OAuth 授权委托和隐私风险 |
| **EXIF 清洗在 App 端** | 发送前本地调用 `@kit.ImageKit` 移除 GPS、设备信息等敏感 EXIF 字段（复用现有 AI feature 中的 EXIF 处理模式） |
| **服务端临时缓存** | trip 元数据 + 转码后 WebP 存入内存/Redis，TTL = 过期时间；过期后 lazy delete 或 cron 批量清理 |
| **URL 由服务端生成** | 服务端决定 shortCode 和签名；App 端不构造 URL |
| **HMAC 密钥仅存服务端** | 签名在 publish 时生成、访问时校验，密钥不出服务器 |
| **H5 访问时动态渲染** | 模板升级后存量分享自动受益；数据内联 HTML 无二次 API 请求 |

## 3. 后端接口契约

> **注意**：自建服务器不在本仓库范围内。此处仅定义 App 端对接的接口契约。

### 3.1 发布分享

**`POST /api/v1/share/publish`**

Content-Type: `multipart/form-data`

| 字段 | 类型 | 说明 |
|------|------|------|
| `tripData` | JSON (text) | 路线元数据 + 节点列表（结构见下） |
| `photos` | File[] (binary) | 照片文件列表，与 `tripData.nodes[].photos` 按顺序对应 |
| `expiryHours` | number | 过期小时数，默认 168（7 天） |

`tripData` JSON 结构：
```json
{
  "tripId": "trip_abc123",
  "tripName": "深圳湾周末骑行",
  "totalDistance": 12.5,
  "coverIndex": 0,
  "createdAt": 1714200000000,
  "nodes": [
    {
      "id": "node_001",
      "title": "红树林公园",
      "content": "下午三点到达，阳光正好...",
      "poiName": "深圳湾红树林",
      "latitude": 22.5231,
      "longitude": 113.9432,
      "photoCount": 2,
      "mood": "惬意",
      "tags": ["骑行", "日落", "自然"],
      "visitedAt": 1714200000000,
      "nodeOrder": 0
    }
  ]
}
```

> **`photoCount`**：每个节点包含的照片数量。`photos[]` 文件列表按 nodeOrder 顺序将所有节点的照片扁平排列，服务端根据 `photoCount` 分组。

响应：
```json
{
  "code": 0,
  "data": {
    "url": "https://tp.app/s/xK9mNz?t=1714800000&s=a7f2c9b3d1e4...",
    "expiresAt": 1714800000
  }
}
```

**服务端处理逻辑**：
1. 解析 multipart：提取 `tripData` JSON + `photos[]` 二进制文件列表
2. 按 `photoCount` 分组，将照片分配到对应节点
3. 照片转码 WebP 两档（375w / 750w）→ 写入临时存储
4. 生成 shortCode（8 位随机字符串，冲突重试）
5. 签名：`HMAC-SHA256(shortCode + ":" + expiryTimestamp, SECRET)` → hex
6. 将 trip 元数据 + 转码后 WebP URL 写入缓存（TTL = expiryHours）
7. 返回签名 URL

### 3.2 H5 页面访问

**`GET /s/:shortCode?t=<expiry>&s=<signature>`**

校验流程：
1. 提取 `shortCode`、`t`、`s` 参数
2. 验证 `HMAC-SHA256(shortCode + ":" + t, SECRET) == s`
3. 验证 `Date.now() / 1000 < t`
4. 从缓存读取 trip 数据
5. 将数据注入 EJS 模板 → 返回完整 HTML

### 3.3 缓存清理策略

```
过期判定: cache.ttl < Date.now()
清理时机:
  - Lazy: GET /s/:shortCode 时发现过期 → 即时删除
  - Cron: 每小时扫描过期条目 → 批量删除（含 WebP 文件）
清理范围: trip 元数据 + 所有关联 WebP 图片文件
```

### 3.4 密钥管理

- `SHARE_HMAC_SECRET` 仅存于服务端（环境变量）
- App 端不持有、不传输密钥
- 签名在 publish 时生成，访问时校验

### 4.3 发布 API 调用

```typescript
import { http } from '@kit.NetworkKit';
import { ApiEndpoints } from '../../../common';

async function publishTrip(
  tripData: ShareTripRequest,
  photos: ArrayBuffer[]
): Promise<string> {
  let httpRequest = http.createHttp();
  let builder = new http.MultipartBuilder();

  // JSON 字段
  builder.addStringPart('tripData', JSON.stringify(tripData));
  builder.addStringPart('expiryHours', '168');

  // 照片二进制字段（扁平排列，与 nodes[].photoCount 对应）
  photos.forEach((data, index) => {
    builder.addBinaryPart(`photos`, data, `photo_${index}.jpg`);
  });

  let response = await httpRequest.request(
    ApiEndpoints.SHARE_PUBLISH,
    {
      method: http.RequestMethod.POST,
      header: { 'Content-Type': 'multipart/form-data' },
      extraData: builder.build()
    }
  );

  let result = JSON.parse(response.result as string);
  return result.data.url;
}
```

### 4.5 安全考量

| 层面 | 措施 |
|------|------|
| **传输安全** | HTTPS 加密所有 publish 请求 |
| **照片隐私** | multipart 传输前 App 端本地 EXIF 清洗（移除 GPS、设备标识） |
| **密钥安全** | App 端不持有 HMAC 密钥，签名完全在服务端完成 |
| **数据最小化** | 仅发送 H5 渲染所需字段，不发送 `syncStatus`、`cloudId`、`ownerUid` 等内部字段 |
| **网络失败** | publish 失败时本地缓存请求数据，用户可手动重试 |

## 7. Phase 2：H5 动态路线回放动画

### 7.1 动画范式（对齐 TripReplayPage）

App 端 `TripReplayPage.ets` 定义的动画节奏是本阶段的**设计基准**：

```
节点 N 周期（7.5s @ 1x）:
┌──────────┬──────────┬──────────┬──────────┐
│ FADE_IN  │ STAYING  │ FADE_OUT │  MOVING  │
│   2.0s   │   3.0s   │   1.0s   │   1.5s   │
│ 卡片渐显 │ 卡片停留 │ 卡片渐隐 │ 相机飞行 │
└──────────┴──────────┴──────────┴──────────┘
                     ↓ 进入下一节点
```

**H5 端必须复刻的动画要素**：

1. **相机飞行**：地图在两个节点坐标之间平滑平移 + 缩放
2. **节点卡片淡入淡出**：opacity 0→1→1→0
3. **路线折线**：polyline 始终可见，当前段高亮
4. **进度指示器**：离散步骤滑块（对应 `ReplayProgressBar`）
5. **速度控制**：0.5x / 1x / 2x

### 7.2 渲染方案

| 方案 | 优势 | 劣势 | 决策 |
|------|------|------|------|
| **PixiJS (WebGL)** | 高性能、相机系统成熟 | 额外依赖 (~200KB) | ✅ Phase 2 |
| Canvas 2D | 原生无依赖 | 相机系统需手写 | ❌ |
| CSS Transform + DOM | 简单 | 复杂地图背景性能差 | ❌ |
| MapLibre GL JS | 真实地图、支持 bearing/tilt | 需瓦片服务 API key | 备选 |

**推荐技术组合**：**PixiJS + GSAP**

- PixiJS：WebGL 渲染层——地图背景纹理 + polyline + 节点标记
- GSAP：`timeline()` 按序编排 FADE_IN → STAYING → FADE_OUT → MOVING
- 地图坐标系：静态高分辨率地图图块作为纹理，相机在该坐标系内平移

**技术参数**：

```
地图坐标系: 4000 × 3000px (虚拟坐标)
视口: ~375 × 812px (设备屏幕)
相机: { x, y, scale } 三元组（GSAP 驱动）
节点坐标: 经纬度 → 虚拟坐标系投影
```

### 7.3 动画状态机

```typescript
// H5 端（对齐 TripReplayPage Phase 枚举）
enum PlayPhase {
  IDLE = 0,
  FADE_IN = 1,   // 2.0s → 卡片 opacity 0→1
  STAYING = 2,   // 3.0s → 卡片保持
  FADE_OUT = 3,  // 1.0s → 卡片 opacity 1→0
  MOVING = 4     // 1.5s → 相机飞行
}

interface PlaybackState {
  currentPhase: PlayPhase;
  currentNodeIndex: number;
  speed: 0.5 | 1 | 2;
  isPlaying: boolean;
  progress: number;  // 0-100
}
```

## 10. 风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| **systemShare 微信不出现** | 用户无法分享到微信 | 备选：复制链接 + 引导用户在微信中粘贴 |
| **大节点量 multipart 超时** | 发布失败 | 前端限制单次分享最多 20 张照片；超时设置 120s（与现有 `HttpTimeout.UPLOAD` 一致） |
| **H5 大图加载慢** | 用户流失 | WebP 双分辨率 + Loading 进度条 |
| **PixiJS WebGL 兼容性** | 低端机白屏 | WebGL 检测 + Canvas 2D 降级 |
| **H5 被微信缓存策略覆盖** | 过期链接仍可访问 | 后端始终校验时间戳；前端倒计时仅供参考 |
| **publish API 调用失败** | 无法生成分享链接 | App 端缓存请求数据，支持手动重试 |
| **服务端缓存膨胀** | 磁盘占满 | 严格 TTL + cron 批量清理；设置缓存上限（如 1000 条） |
